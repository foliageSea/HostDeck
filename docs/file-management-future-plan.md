# 文件管理扩展方向分析

## 背景

当前文件管理页面已经具备基础可用能力，主要包括：

- 目录浏览与上下级导航
- 列表 / 网格视图切换
- 单选、多选、框选
- 右键菜单操作
- 上传、下载、复制、剪切、粘贴
- 新建目录、重命名、删除
- 收藏目录
- 文本文件打开编辑器
- 图片 / 视频文件打开预览器
- 在当前目录打开终端

从实现结构看，当前模块已经具备继续扩展的良好基础：

- `ssh-tool-ui/src/views/Files.vue` 主要负责页面装配
- `ssh-tool-ui/src/stores/file.ts` 集中文件管理状态
- `ssh-tool-ui/src/composables/file/useFileOperations.ts` 封装核心文件操作
- `ssh-tool-ui/src/composables/file/useFileContextMenu.ts` 负责右键菜单逻辑
- `ssh-tool-ui/src/composables/file/useFileSelection.ts` 负责框选交互

整体上，当前实现已经形成了“页面装配 + store 状态 + composable 行为”的清晰分层。后续扩展可以继续沿着这条结构演进，而不需要推翻现有代码。

---

## 当前结构判断

### 1. 页面层较薄，适合继续扩能力

`Files.vue` 本身没有承载太多业务逻辑，主要负责：

- 挂载工具栏、列表 / 网格组件、右键菜单、弹窗、上传进度
- 初始化 `fileStore`
- 组合 `useFileSelection`、`useFileOperations`、`useFileContextMenu`

这意味着新增功能时，大多数改动都可以放在 store 或 composable 中，页面本身只做接线。

### 2. Store 已经具备“多实例扩展”的雏形

`file.ts` 中已经存在：

- 全局剪贴板 `globalClipboard`
- 全局刷新信号 `refreshSignal`
- 收藏目录 `favorites`

这说明当前设计已经部分考虑了多个文件管理实例共享状态的场景，例如：

- 多窗口文件管理器
- 双栏模式
- 多标签页文件管理

这是未来继续做复杂工作区能力的重要基础。

### 3. 文件操作集中，扩展效率高

`useFileOperations.ts` 目前已经是文件管理动作的核心入口。新能力如果属于“用户对文件执行一个动作”，通常都可以顺着这里继续扩展，例如：

- 属性查看
- 新建文件
- 权限修改
- 压缩 / 解压
- 批量重命名

这一点对后续迭代很有利。

### 4. 当前也存在两个潜在风险

#### 风险一：`useFileOperations.ts` 继续增长后容易变胖

目前它已经覆盖上传、下载、重命名、删除、打开、终端联动等多类能力。继续叠加功能后，文件可能迅速膨胀，影响维护性。

短期内问题不大，但中后期可以考虑按领域拆分，例如：

- `useFileTransferOperations`
- `useFileMutationOperations`
- `useFileOpenOperations`

#### 风险二：`fileStore` 既管理状态，又承载部分业务副作用

当前 store 中除了状态，还直接包含：

- 会话初始化
- 列表拉取
- 复制 / 剪切状态写入
- 刷新通知

这在当前规模下是高性价比方案，但如果继续加入历史、缓存、搜索、分页、任务队列，store 可能逐渐变成过重的业务中心。后续应尽量保持：

- store 负责状态与轻量状态变换
- 重副作用逻辑优先放在 composable 或 service

## 主要扩展方向

## 1. 导航体验增强

### 当前状态

当前已有导航能力主要是：

- 当前路径 `currentPath`
- 进入目录 `navigate()`
- 返回上级 `navigateUp()`
- 刷新 `refresh()`

这足够支撑基础浏览，但还没有形成成熟文件管理器常见的导航体验。

### 可扩展内容

- 前进 / 后退历史栈
- 面包屑导航
- 路径输入跳转
- 最近访问目录
- 收藏目录侧边栏
- 常用目录快捷入口
- 根目录 / Home / 临时目录快速跳转

### 价值判断

这是最值得优先做的一类能力，因为：

- 与当前 `currentPath` 中心化模型天然契合
- 对现有后端 API 几乎无侵入
- 用户体感提升非常直接

### 建议落点

- `fileStore` 增加：`backHistory`、`forwardHistory`、`recentPaths`
- `FileToolbar` 增加：前进 / 后退按钮、面包屑、路径输入框
- 需要更完整结构时，可新增 `FileSidebar.vue`

---

## 2. 排序、筛选与搜索能力

### 当前状态

当前文件列表逻辑较简单：

- 过滤 `.` 和 `..`
- 按“目录优先 + 文件名升序”排序

说明数据展示能力还有很大扩展空间。

### 可扩展内容

- 排序字段切换：
  - 名称
  - 大小
  - 修改时间
  - 类型
- 排序方向切换：
  - 升序
  - 降序
- 前端即时过滤：
  - 文件名模糊搜索
  - 仅目录
  - 仅文件
  - 仅图片 / 媒体 / 代码文件
  - 显示 / 隐藏隐藏文件
- 递归搜索：
  - 当前目录内搜索
  - 跨子目录搜索

### 分阶段建议

第一阶段可只做前端本地排序和过滤，基于已有 `files` 列表即时计算。

第二阶段若要支持大目录或递归搜索，再考虑让后端接口支持：

- server-side search
- server-side sort
- 分页或增量加载

### 价值判断

这一方向属于典型高频能力，投入小、收益高，适合作为紧随导航优化之后的下一步。

---

## 3. 文件操作能力补全

### 当前状态

当前已支持：

- 下载 / 批量下载
- 上传 / 拖拽上传
- 复制 / 剪切 / 粘贴
- 重命名
- 新建目录
- 删除
- 打开文件 / 目录

对于一个远程 SSH 文件管理器来说，这已经形成了基础闭环，但离“更完整的文件管理能力”还有差距。

### 值得补齐的能力

- 新建文件
- 查看属性
- 权限修改（如 `chmod`）
- 所有者修改（如 `chown`，若后端权限允许）
- 压缩 / 解压
- 批量重命名
- 复制为 / 移动到
- 软链接查看与处理
- 文件哈希查看（MD5 / SHA256）
- 文本文件快速预览
- 目录下载格式选择

### 优先建议

优先级较高的几个点：

- 新建文件
- 属性面板
- 权限修改
- 批量重命名

原因是它们都很贴近远程运维 / 开发场景中的日常需求，而且与当前菜单和弹窗体系非常匹配。

### 建议落点

- 动作实现：`useFileOperations.ts`
- 菜单入口：`useFileContextMenu.ts`
- 弹窗承载：`FileModals.vue` 或新增独立属性弹窗组件

---

## 4. 选择与交互模型增强

### 当前状态

目前已有：

- 单选
- Ctrl / Cmd 多选
- 全选
- 鼠标框选

基础交互是成立的，但与成熟文件管理器相比还偏轻量。

### 可扩展内容

- Shift 连续范围选择
- 键盘导航：
  - 方向键移动焦点
  - Enter 打开
  - Delete 删除
  - F2 重命名
  - Ctrl / Cmd + C / X / V
- 双击 / 单击打开策略可配置
- 拖拽移动到目录
- 拖拽到收藏目录
- 触屏模式适配

### 结构提醒

当前 `selectedFiles` 使用的是 `Set<string>`，仅保存文件名。这对当前“当前目录内选择文件”的场景是足够的，但未来如果引入以下能力，会逐渐暴露限制：

- 跨目录搜索结果统一展示
- 多 pane / 多标签并存
- 拖拽跨区域移动
- 虚拟列表场景下的稳定项标识

届时更稳妥的做法可能是把选中项标识升级为完整路径或稳定 ID。

---

## 5. 多视图与工作区能力

### 当前状态

当前视图模式只有：

- `list`
- `grid`

这是一种很好的最小实现，但仍有进一步扩展的空间。

### 可扩展内容

- 双栏文件管理器
- 多标签页
- 分屏目录对比
- 左侧侧边栏 + 主区布局
- 收藏夹 / 最近访问 / 远端挂载点展示
- 文件任务面板
- 关闭后恢复会话

### 价值判断

这一方向更偏“产品升级”，适合在基础浏览体验和核心文件操作稳定后推进。

值得注意的是，当前全局剪贴板和全局刷新机制已经为多实例协作打下了一部分基础，但如果实例数量增加，后续可能需要更细粒度的刷新事件模型，例如：

- 按 `sessionId` 刷新
- 按路径前缀广播变更
- 基于 mutation event 的明确通知机制

---

## 6. 统一任务系统与操作反馈

### 当前状态

当前长操作反馈主要有两类：

- 页面级 `loading`
- 上传进度 `uploadStatus`

这对于上传场景是够用的，但对于批量复制、批量删除、大目录下载等耗时任务仍然不够完整。

### 可扩展内容

- 通用文件任务队列
- 每个任务具备：
  - `id`
  - `type`
  - `status`
  - `progress`
  - `currentItem`
  - `error`
- 支持查看任务详情
- 支持取消任务（若后端支持）
- 支持失败重试
- 支持历史任务记录

### 价值判断

远程文件管理的核心特征之一就是操作延迟较高，因此统一任务反馈系统对用户体验提升非常明显。它能显著增强“操作正在进行中”的确定感，也是后续复杂能力的基础设施。

### 建议落点

- 在 store 中增加统一任务状态
- 将 `FileUploadProgress` 逐步演进为更通用的 `FileTaskPanel`

---

## 7. 与编辑器 / 终端的协同深化

### 当前状态

当前模块已经与桌面工作台形成了初步联动：

- 文本文件可在编辑器打开
- 图片 / 视频可在媒体查看器打开
- 当前目录可直接在终端中打开

这说明文件管理器并不是孤立页面，而是整个远程工作台的重要入口。

### 可扩展内容

- “打开方式”选择
- 默认打开方式配置
- 编辑器保存后自动刷新目录
- 终端内目录切换与文件管理器同步
- 右键菜单增加“在编辑器中打开”“在终端中定位”
- 最近打开文件 / 最近目录联动
- 文件变更后主动提示刷新

### 价值判断

这一方向的价值不只是“多一个动作”，而是把文件管理模块从独立工具升级为工作流枢纽。

---

## 8. 远程环境语义与元信息增强

### 当前状态

当前 `FileItem` 暴露的字段较少：

- `filename`
- `longname`
- `isDirectory`
- `size`
- `modifyTime`

其中 `longname` 暗示后端可能已经拿到了更丰富的远程文件信息，但当前前端并没有把这些信息结构化消费起来。

### 可扩展内容

- 权限位解析
- 所有者 / 所属组展示
- 链接目标展示
- 可执行文件识别
- 特殊文件类型识别：
  - socket
  - fifo
  - device
- 更准确的文件图标和文件类型标签

### 价值判断

SSH 文件管理器与本地文件管理器最大的差异之一，就是远程 Unix 环境语义更强。补齐这些元信息，能明显提升工具的专业度，也能为属性面板、权限编辑、打开方式识别提供基础。

---

## 9. 大目录性能与可扩展性

### 当前状态

当前模型基本是：

- 一次性拉取整个目录列表
- 前端排序和渲染全部文件
- 框选时缓存所有文件节点的 `getBoundingClientRect()`

在中小规模目录下问题不大，但当目录内容达到数千甚至数万时，性能问题会逐渐显现。

### 可能遇到的瓶颈

- 列表渲染卡顿
- 过滤 / 搜索实时计算卡顿
- 框选命中成本升高
- 右键选择时状态更新变重
- 网格模式在超大目录下更容易卡顿

### 建议方向

- 列表虚拟滚动
- 大目录自动切换更轻量视图
- 分页或增量加载
- 后端支持排序 / 搜索 / 分页
- 对框选算法做按可见区域优化

这一方向可以先不急着做，但如果后续要支持大规模服务器目录或日志目录浏览，就需要提前纳入设计考虑。

---

## 建议优先级

从投入产出比来看，建议优先级如下：

1. 导航体验增强
2. 排序、筛选与搜索
3. 新建文件、属性面板、权限修改
4. 键盘快捷键与范围选择
5. 统一任务系统
6. 双栏 / 多标签 / 工作区能力
7. 大目录性能优化
8. 更完整的远程元信息支持

### 优先级说明

- 前 1 到 4 项都是高频刚需，并且与现有架构天然兼容
- 第 5 项属于中期基础设施建设，会显著提升复杂操作体验
- 第 6 到 8 项更像产品能力升级，适合在基础体验稳定后推进

---

## 推荐的下一步最小迭代包

如果要基于当前实现快速推进一轮高收益迭代，建议优先做以下组合：

1. `FileToolbar` 增加面包屑、前进后退、搜索框
2. `fileStore` 增加排序 / 筛选状态
3. `useFileContextMenu` 增加“新建文件”“属性”入口
4. `FileModals` 增加“新建文件”与“属性”相关弹窗
5. 文件列表支持 `Shift` 范围选择和基础快捷键

这组改动有几个优点：

- 不需要大动后端接口
- 不会破坏当前分层结构
- 能明显提升文件管理器的完整度与专业感

---

## 总结

当前文件管理模块的最大优势，不是“功能已经很多”，而是“结构已经足够适合继续长功能”。

它已经具备以下几个很好的扩展前提：

- 页面足够薄，便于继续装配新能力
- store 已具备中心化状态管理能力
- 文件动作集中在 composable 中，便于新增行为
- 与编辑器、终端、媒体查看器已有初步联动

因此，后续扩展最适合沿着以下节奏推进：

- 先补高频体验能力：导航、筛选、搜索、属性、快捷键
- 再补复杂操作基础设施：任务系统、批量操作、权限语义
- 最后再做产品级升级：双栏、多标签、大目录优化、工作区协同

如果后续要进入实现阶段，建议先从“导航增强 + 搜索排序 + 属性面板”这个组合开始，成本低、感知强，也最符合当前代码结构。
