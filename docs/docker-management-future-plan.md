# Docker 管理 Future 计划

## 背景

当前 Docker 管理模块已具备基础能力：

- 检查远程服务器 Docker 可用性
- 查看容器列表与镜像列表
- 启动、停止、重启、删除容器
- 删除镜像
- 查看容器日志
- 进入容器 Shell

整体已形成最小可用闭环，但当前仍以“基础运维动作”为主，距离更完整的远程容器管理能力还有较大扩展空间。

## 总体目标

将 Docker 管理模块逐步升级为具备以下能力的远程运维面板：

- 更高效的列表浏览与筛选
- 更强的日志与排障能力
- 更完整的容器详情与资源监控
- 更实用的镜像与容器生命周期管理
- 项目级的 Docker Compose 管理
- 与终端、文件管理等现有模块深度联动

---

## Phase 1：基础体验增强

### 1. 容器列表增强

**目标**：提升容器数量较多场景下的可用性

**计划内容**

- 增加容器名称搜索
- 增加镜像名称搜索
- 增加状态过滤：
  - `全部`
  - `running`
  - `exited`
  - `paused`
- 增加排序方式：
  - 创建时间
  - 名称
  - 镜像
- 支持点击统计卡片快速筛选

**预期收益**

- 快速定位目标容器
- 降低大规模服务器下的操作成本

### 2. 日志能力增强

**目标**：提升日志查看与问题排查效率

**计划内容**

- 支持切换日志行数：
  - `100`
  - `200`
  - `500`
  - `1000`
- 支持日志手动刷新
- 支持日志自动刷新
- 支持日志关键字搜索
- 支持显示或隐藏时间戳
- 支持复制日志
- 支持下载日志内容

**预期收益**

- 当前日志弹窗从“只读查看”升级为“轻量排障工具”

### 3. 镜像列表增强

**目标**：提升镜像清理和管理效率

**计划内容**

- 增加仓库名或标签搜索
- 增加排序能力
- 支持识别悬空镜像（dangling images）
- 支持识别未使用镜像
- 支持批量清理无引用镜像

**预期收益**

- 减少磁盘占用
- 提升镜像管理可操作性

### 4. 批量操作

**目标**：覆盖常见批量运维场景

**计划内容**

- 批量启动容器
- 批量停止容器
- 批量删除已停止容器
- 一键清理 dangling images

**预期收益**

- 适合测试环境、开发环境、批量维护场景

---

## Phase 2：排障与可观测能力增强

### 1. 容器详情面板

**目标**：从“列表操作”扩展到“单容器诊断”

**计划内容**

- 新增容器详情抽屉或弹窗
- 展示以下信息：
  - 完整容器 ID
  - 完整状态
  - 镜像信息
  - 启动命令
  - 端口映射
  - 挂载卷
  - 网络信息
  - 环境变量
  - Labels
  - Restart Policy
  - 健康检查状态

**后端依赖**

- 新增 `docker inspect` 对应接口

**预期收益**

- 避免用户反复切终端执行 inspect 命令
- 提升问题定位效率

### 2. 容器资源监控

**目标**：补齐运行态可观测能力

**计划内容**

- 基于 `docker stats --no-stream` 展示：
  - CPU 使用率
  - 内存使用率
  - 网络 IO
  - 磁盘 IO
  - PIDs
- 支持列表中快速展示简版资源信息
- 支持详情中查看完整资源数据
- 支持定时刷新

**预期收益**

- 更适合监控异常容器
- 为后续告警和健康分析打基础

### 3. 异常状态识别

**目标**：提升异常容器可见性

**计划内容**

- 高亮异常退出容器
- 标记频繁重启容器
- 标记 `unhealthy` 容器
- 提供“快速查看最近日志”入口

**预期收益**

- 从“被动查看”变为“主动发现异常”

---

## Phase 3：镜像与容器管理能力补齐

### 1. 镜像管理扩展

**目标**：让镜像页具备基础完整性

**计划内容**

- 拉取镜像
- 镜像重新打标签
- 查看镜像历史
- 查看镜像被哪些容器引用
- 清理未使用镜像

**预期收益**

- 补齐从镜像到容器的常用链路
- 减少用户依赖终端执行基础镜像操作

### 2. 容器创建能力

**目标**：支持从 UI 直接创建容器

**计划内容**

- 增加“新建容器”表单
- 支持配置：
  - 镜像
  - 容器名称
  - 端口映射
  - 环境变量
  - 卷挂载
  - 后台运行
  - Restart Policy

**后续可扩展**

- 从现有容器复制配置创建
- 保存为模板

**预期收益**

- 从“管理已有容器”升级为“完整生命周期管理”

### 3. 容器操作补充

**目标**：补足高频操作闭环

**计划内容**

- Pause / Unpause
- Rename
- 查看原始 inspect JSON
- 导出容器配置
- 快速重建容器

**预期收益**

- 满足更细粒度的容器维护需求

---

## Phase 4：Docker Compose / 项目级管理

### 1. Compose 项目管理

**目标**：从单容器视角升级到项目视角

**计划内容**

- 展示 Compose 项目列表
- 展示项目下服务与容器
- 支持操作：
  - `up`
  - `down`
  - `restart`
  - `pull`
  - `logs`
- 支持查看 `docker-compose.yml`
- 支持与文件管理器联动打开配置文件

**预期收益**

- 更符合真实服务器上的使用场景
- 让 Docker 管理不止停留在单容器层面

### 2. 项目视图增强

**目标**：增强项目级可视化和操作效率

**计划内容**

- 按项目聚合容器
- 展示服务状态
- 展示依赖关系
- 支持项目级日志查看
- 支持一键重建服务

**预期收益**

- 提升复杂应用栈管理体验

---

## Phase 5：深度联动与产品化增强

### 1. 与现有模块联动

**目标**：发挥当前应用“SSH + 文件 + 终端”的组合优势

**计划内容**

- 从容器详情直接进入 Shell
- 支持容器内文件浏览
- 支持 `docker cp` 上传或下载文件
- 快速打开容器相关配置目录
- 与终端窗口联动执行高频操作

**预期收益**

- 强化产品整体一致性
- 降低跨模块切换成本

### 2. 自动化运维动作

**目标**：沉淀高频操作模板

**计划内容**

- 拉取新镜像并重建容器
- 重启容器后自动打开日志
- 导出容器运行快照
- 一键清理无用资源：
  - 容器
  - 镜像
  - 网络
  - 卷

**预期收益**

- 从基础工具向“运维效率工具”演进

### 3. 安全与审计

**目标**：提升危险操作可控性

**计划内容**

- 危险操作增加二次确认
- 删除前显示影响范围
- 支持只读模式
- 记录敏感操作日志

**预期收益**

- 更适合生产环境使用
- 降低误操作风险

## 后端配套计划

### 1. 新增接口能力

建议逐步补充以下接口：

- `docker inspect`
- `docker stats --no-stream`
- `docker pull`
- `docker image history`
- `docker compose ls`
- `docker compose ps`
- `docker compose logs`
- `docker compose pull`
- `docker compose up/down`

### 2. 参数安全治理

当前 Docker 服务大多通过 SSH 执行 Docker CLI，后续扩展时建议：

- 统一对容器 ID、容器名、镜像名、tag、compose project 做 shell quote
- 避免后续新增命令出现命令拼接风险

### 3. 接口分层策略

建议保持以下原则：

- 列表接口尽量轻量
- 详情接口按需查询
- 日志与资源监控优先采用轮询
- 长连接或流式能力后续再评估

包含：

- Docker Compose 管理

## 结论

Docker 模块的下一阶段建议优先围绕“排障效率”和“运维效率”展开，而不是一开始就做过重的编排功能。

最优先落地的方向是：

1. 列表搜索过滤
2. 日志增强
3. 容器详情
4. 资源监控

这样既贴合当前架构，也能最快提升实际使用价值。
